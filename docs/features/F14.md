## Purpose

Enables the organization of users into collaborative units called workspaces. Workspaces provide a directory of members that acts as a default recipient pool for email summaries, ensuring that team members stay informed about meeting outcomes even if they are not explicitly selected for a specific meeting.

## User Journey

1. The user navigates to the Settings view in the application.
2. The user creates a new Workspace by providing a name.
3. The user adds members to the workspace by specifying their names and email addresses.
4. When creating a new meeting, the user can optionally link it to this workspace.
5. If an email summary is dispatched for that meeting and no specific recipients were selected, the system automatically retrieves the workspace members and sends the summary to all of them.
## Functional Behaviour

The system provides standard CRUD operations for managing `Workspace` entities, which contain a name and a list of member objects `{name, email}`. A meeting can optionally reference a workspace ID. During post-meeting processing, if the `email_summary` automation needs a recipient list and the meeting's `selectedRecipients` array is empty, the system looks up the associated workspace and uses its member list as the default recipient pool.

## Inputs

| Input | Source | Description |
|-------|--------|-------------|

## Processing

1. **CRUD**: Standard Express controller handling create, read, update, delete requests for the `Workspace` collection in MongoDB.
2. **Member Resolution**: When an email summary is triggered, if explicit recipients are missing, query the `Workspace` collection using the meeting's `workspaceId` to retrieve default members.
