## Purpose

Provides a human-in-the-loop safety mechanism for AI-detected actions. When the system identifies a user intent (e.g., scheduling a meeting or sending an email), it does not execute immediately. Instead, it creates a pending task that the user can review, modify parameters for, and then approve or reject. This prevents unintended side effects from misinterpreted voice commands.

## User Journey

1. During a live meeting, a voice command is detected and classified (e.g., `schedule_meeting`).
2. The system refines the raw command text into structured parameters using the LLM (e.g., extracting a title, date, and participants).
3. A pending automation card appears in the UI sidebar/notification area, showing the detected intent, extracted parameters, and a confidence score.
4. The user reviews the automation. They can edit parameters (e.g., change the meeting time) before approving.
5. Upon clicking "Approve," the system dispatches the action to the appropriate handler (M1 for meetings, M10 for emails).
6. Alternatively, the user clicks "Reject" to dismiss the automation entirely.
7. The status transitions to `completed` or `rejected` and the card is removed from the pending list.
## Functional Behaviour

The Automation Orchestrator manages a state machine for each automation log: `pending â†’ approved â†’ completed/failed` or `pending â†’ rejected`. On intent ingestion, the system first checks for duplicate active logs of the same intent for the same meeting to enforce idempotency. If no duplicate exists, it calls the Intelligence Core (M5) to refine the raw command text into structured parameters (e.g., extracting a date from "next Tuesday at 3 PM"), assigns a confidence score, and persists the `AutomationLog` with status `pending`. The frontend polls the `GET /api/automation/pending` endpoint to display pending items. On approval, the orchestrator updates the status, records any user-edited parameters, and dispatches execution based on the intent type. For `schedule_meeting`, it creates a new meeting internally. For `email_summary`, it generates an HTML email body and dispatches to an n8n webhook. For `create_visualization`, it auto-executes without requiring approval. On rejection, the status transitions directly to `rejected`.

## Processing

1. **Ingest**: Receive classified intent â†’ Check for existing active duplicates â†’ Call M5 to refine parameters â†’ Persist `AutomationLog` with status `pending`.
2. **List Pending**: Query `AutomationLog` where status = `pending`, sorted by creation date descending â†’ Transform and return.
3. **Approve**: Find log by ID â†’ Update status to `approved` â†’ Store edited parameters â†’ Route to intent handler:
   - `schedule_meeting` â†’ Create a new meeting via M1's repository
   - `email_summary` â†’ Generate HTML body, resolve recipients, dispatch webhook to n8n via M10
   - `create_visualization` â†’ Call M7 to generate chart
4. **Reject**: Find log by ID â†’ Update status to `rejected`.

## Limitations

- Adding a new intent type requires modifying brittle switch/if logic in the orchestrator.
- The orchestrator directly contains the execution logic for meeting creation and email HTML generation instead of delegating cleanly.
- No retry mechanism for failed webhook dispatches.
- No configurable auto-approval rules based on confidence thresholds.
- Visualization intents auto-execute without going through the approval workflow, inconsistent with other intents.
- No audit trail UI â€” users cannot see historical approved/rejected automations.
