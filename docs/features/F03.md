## Purpose

Enables the system to run a fully functional demo meeting without requiring an actual audio source or Chrome extension. A scripted, timed transcript is injected into the ingestion pipeline, triggering all downstream features (command detection, visualization, insights) exactly as they would during a real meeting. This is essential for demonstrations, testing, and development.

## User Journey

1. The user creates a new meeting in the NeuroNotes UI.
2. The user triggers the simulation endpoint (via API call or a UI action).
3. The system begins injecting scripted transcript segments at timed intervals, simulating a realistic meeting conversation.
4. The Live Meeting view shows transcript appearing in real-time, with speaker labels and timestamps.
5. All downstream features activate as if the meeting were real â€” voice commands are detected, visualizations may be generated, and the chat/voice assistants become usable with the simulated context.
6. The user ends the meeting normally to complete the simulation.
## Functional Behaviour

The SimulationService contains a pre-scripted transcript with speaker labels and timed delays. When triggered, it iterates through the script, calling the internal ingestion logic at scheduled intervals to mimic the cadence of a real conversation. Each injected segment passes through the same pipeline as real webhook data â€” speaker resolution, MinuteWindow storage, Firestore sync, and command interpreter routing. The simulation is tied to a specific meeting ID and runs in the background using timers.

## Processing

1. **Trigger**: Receive the start-simulation request with a meeting ID.
2. **Load Script**: Retrieve the pre-defined transcript script (hardcoded in the service).
3. **Timed Injection**: For each line in the script, wait the specified delay, then call the internal ingestion function with the segment data (text, speaker, calculated timestamp).
4. **Standard Pipeline**: Each injected segment flows through the same pipeline as real data â€” speaker resolution, MinuteWindow upsert, Firestore write, command detection.
5. **Complete**: The simulation ends when all scripted lines have been injected.

## Limitations

- The simulation script is hardcoded; users cannot define custom simulation scenarios.
- Only one simulation can run per meeting at a time.
- Simulation service logic is intertwined with production ingestion controllers rather than being cleanly isolated.
- No mechanism to pause or adjust simulation speed.
