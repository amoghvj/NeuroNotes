## Purpose

Captures and stores real-time speech-to-text data during a live meeting. This provides the raw conversational data that powers every downstream AI feature â€” summaries, insights, visualizations, chat, and voice interaction all depend on the transcript being available and up-to-date.

## User Journey

1. The user starts a meeting in NeuroNotes and activates the Transcriptonic Chrome Extension on their Google Meet call.
2. The Chrome extension captures live audio transcription and sends batches of transcript segments to the NeuroNotes backend via a webhook endpoint.
3. The transcript appears in the Live Meeting view in near real-time, with speaker labels and timestamps.
4. As the meeting progresses, the transcript automatically scrolls (with auto-scroll toggle) and highlights speaker changes.
5. After the meeting ends, the full transcript remains available for review in the meeting detail view.
## Functional Behaviour

The Transcript Engine accepts incoming speech-to-text data through two ingestion paths: webhook batch (from the Chrome extension) and individual chunk ingestion. Each incoming segment contains raw text, a speaker identifier, and a timestamp. The system resolves generic speaker identifiers (e.g., "0", "Unknown") to participant names using the active meeting's participant list â€” defaulting the first participant or "Host" for the primary speaker. Segments are grouped into 1-minute time windows (`MinuteWindow` documents) based on their timestamps, with each window storing an array of segments alongside legacy flat fields. Data is persisted to Firestore for real-time syncing to the frontend. Upon ingestion, each chunk is also passed to the Command Interpreter (M3 via AutomationService) for voice command detection.

## Inputs

| Input | Source | Description |
|-------|--------|-------------|
| Transcript Batch | Transcriptonic Chrome Extension (POST webhook) | Array of objects containing text, speaker identifier, and timestamp |
| Individual Chunk | Direct API call (POST) | Single transcript segment with meetingId, text, speaker, timestamp |
| Meeting Context | M1 (Meeting Lifecycle) | Active meeting ID, participant list for speaker resolution |

## Processing

1. **Receive**: Accept incoming transcript data via `/api/ingest/webhook` or `/api/ingest/chunk`.
2. **Validate**: Confirm a live meeting exists and extract participant list.
3. **Resolve Speaker**: Map the raw speaker identifier to a named participant using heuristics (first participant = host, numbered IDs map to participant array indices).
4. **Time-Bucket**: Calculate which 1-minute window the segment belongs to based on its timestamp. Upsert the `MinuteWindow` document, appending the segment to the `segments` array.
5. **Persist**: Write to Firestore (for real-time frontend updates) and/or MongoDB.
6. **Route to Command Detection**: Pass the chunk text and speaker to `AutomationService.processChunk()` for intent detection.
7. **Respond**: Return success confirmation with processed count to the webhook caller.
## Outputs

| Output | Description |
|--------|-------------|
| MinuteWindow Documents | Time-bucketed transcript segments with resolved speakers |
| Success Response | `{ success: true, processedCount: N }` to the webhook caller |
| Real-time UI Update | Firestore writes trigger live transcript rendering in the frontend |

## Limitations

- Speaker resolution uses basic heuristics (array index mapping); no ML-based speaker diarization.
- The webhook handler synchronously calls automation/intent detection before responding, risking timeouts if downstream processing is slow.
- Speaker resolution logic is duplicated across multiple locations in the codebase.
- No support for multiple simultaneous transcription providers.
- No audio stream ingestion â€” the system only accepts pre-transcribed text.
- No idempotency token on webhook payloads; replayed webhooks could result in duplicate segments.
