## Purpose

Enables hands-free, spoken interaction with the meeting's AI assistant. Users speak a question and receive both a text answer and synthesized audio playback, allowing them to query meeting intelligence without typing. This is the audio counterpart to the text-based AI Chat Assistant (F10).

## User Journey

1. The user clicks the microphone button in the meeting view to activate voice mode.
2. The browser captures the user's spoken question via the Web Speech API (client-side STT).
3. The transcribed text is sent to the backend along with the meeting ID.
4. The system generates a speech-optimized text response (concise, no markdown) grounded in the transcript.
5. The text is converted to audio via the ElevenLabs TTS API.
6. The response plays back as audio in the browser, and the text is also displayed on screen.
7. If the TTS service is unavailable, the text response is still displayed without audio.
## Functional Behaviour

The voice interaction endpoint receives a meeting ID and a text query (already transcribed client-side). It fetches transcript context from M2, constructs a speech-optimized prompt that explicitly forbids markdown formatting (asterisks, hash symbols, code blocks) and requests concise 2-3 sentence answers. The prompt is sent to M5 (Intelligence Core) to generate the text response. The text response is then passed to `VoiceService.generateAudio`, which calls the ElevenLabs TTS API (`api.elevenlabs.io/v1/text-to-speech/{voiceId}`) with configured parameters (stability, similarity boost). The audio buffer is encoded to base64 for JSON transport. If the ElevenLabs API key is missing or the API call fails, a hardcoded silent/dummy MP3 buffer is returned so the frontend doesn't crash. The combined `{ text, audio }` payload is returned to the client.

## Limitations

- Serial latency: the system waits for the LLM response to complete before starting TTS synthesis â€” these are not parallelized.
- Audio is transported as base64 in a JSON payload over REST â€” inefficient and prevents streaming playback.
- The speech-optimized prompt is currently hardcoded inside `LLMService` rather than being owned by the Voice Interaction module.
- Client-side STT (Web Speech API) accuracy varies across browsers and may misinterpret domain-specific terms.
- Only one TTS voice is supported (configured via environment variable); no voice selection UI.
- TTS API quotas and costs are not monitored or exposed to the user.
