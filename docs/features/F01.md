## Purpose

Allows users to create, track, and manage meeting sessions. Meetings serve as the foundational anchor entity for all other system features â€” transcripts, insights, visualizations, and automations are all scoped to a meeting ID. Without this feature, no other capability in NeuroNotes can function.

## User Journey

1. The user opens the NeuroNotes dashboard and clicks "New Meeting."
2. A modal appears where they enter a title, select participants, optionally attach a workspace, provide a meeting link, and choose email recipients.
3. Upon submission, a new meeting record is created with status `live` and the user is navigated to the Live Meeting view.
4. During the meeting, the elapsed timer tracks duration and participants are displayed.
5. When the meeting concludes, the user clicks "End Meeting."
6. The meeting transitions to `completed` status, the end time is recorded, and post-meeting processing (summary generation, email automation) is triggered asynchronously.
7. The completed meeting appears in the Meetings History view, where users can review it or delete it.
## Functional Behaviour

The system manages meetings through a strict state machine: `scheduled â†’ live â†’ completed`. Forward-only transitions are enforced â€” a completed meeting cannot revert to live. The meeting entity stores metadata (title, participants, meeting link, workspace reference, selected recipients) and accumulates a summary object upon completion. Meeting creation defaults the status to `live` unless overridden. The "End Meeting" action updates the status to `completed`, sets the `endTime`, and triggers asynchronous post-meeting workflows (summary generation via M6, email automation creation via M4). Meeting deletion is a hard delete that removes the record. Listing meetings returns all meetings sorted by creation time, with data transformed to match the frontend TypeScript interface via the `transformMeeting` utility.

## Processing

1. **Create**: Validate input fields â†’ Insert `Meeting` document into MongoDB with status `live` and current timestamp as `startTime` â†’ Return transformed meeting object.
2. **List**: Query MongoDB for all meetings â†’ Sort by creation date descending â†’ Transform each document for frontend compatibility â†’ Return array.
3. **Get by ID**: Query MongoDB by ID â†’ Transform document â†’ Return single meeting object, or 404.
4. **End Meeting**: Find meeting by ID â†’ Update `status` to `completed` and set `endTime` â†’ Return success to client immediately â†’ Asynchronously: fetch workspace members for recipient resolution, retrieve full transcript, call Insight Generator for summary, save summary to meeting, create pending `email_summary` AutomationLog entry.

## Limitations

- No recurring meeting support; each meeting is a one-off entity.
- No calendar integration (Google/Outlook) for automatic scheduling or sync.
- No role-based participant permissions (all participants are equal).
- Meeting deletion is a hard delete with no archive or soft-delete mechanism.
- The "End Meeting" action currently performs synchronous post-processing inline, risking failure of the meeting state transition if downstream services (LLM, webhook) are unavailable.
- No zombie meeting cleanup â€” meetings left in `live` status indefinitely are not automatically resolved.
