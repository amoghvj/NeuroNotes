## Purpose

Allows meeting participants to issue commands to the NeuroNotes system by speaking during a meeting. By saying "Hey NeuroNotes" to start and "Over" to finish, users can request actions like scheduling a follow-up meeting or generating a visualization â€” without leaving the meeting or using the UI.

## User Journey

1. During a live meeting with active transcript capture, a participant says: "Hey NeuroNotes, schedule a follow-up meeting for next Tuesday at 3 PM. Over."
2. The system detects the wake word "Hey NeuroNotes" in the transcript stream and begins buffering the spoken text.
3. When "Over" is detected, the buffer is closed and the captured text is normalized (filler words removed, punctuation stripped).
4. The system classifies the intent as `schedule_meeting` based on keyword matching.
5. The classified intent is passed to the Automation Orchestrator for parameter refinement and approval workflow.
6. The user sees a pending automation card appear in the UI.
## Functional Behaviour

Every ingested transcript chunk is evaluated against the wake word pattern (Regex: `/hey[\s,.\-!]*neuro/i`). When detected, a buffer is opened for that meeting ID, capturing all subsequent text until the end marker (`/over/i`) is found. The system maintains one buffer per meeting in an in-memory `Map`. Upon buffer close, the captured text undergoes normalization: lowercasing, punctuation removal, and stripping of defined filler words (e.g., "um", "uh", "like"). The normalized text is then classified against intent categories using keyword arrays â€” `schedulingKeywords` map to `schedule_meeting`, `visualizationKeywords` map to `create_visualization`. The speaker who triggered the wake word is preserved as the command owner. The structured intent payload is then routed to the Automation Orchestrator for further processing.

## Processing

1. **Wake Word Scan**: Check incoming text against the wake word Regex pattern.
2. **Buffer Management**: If wake word detected and no buffer exists, open a new buffer for this meeting ID, recording the speaker. If a buffer is already open, append the text.
3. **End Marker Scan**: Check for the "Over" end marker in each chunk while a buffer is active.
4. **Buffer Close**: When "Over" is detected, extract the complete command block from the buffer and delete the buffer entry.
5. **Normalize**: Apply lowercasing, punctuation removal, and filler word stripping to the raw command text.
6. **Classify**: Match normalized text against intent keyword arrays to determine `IntentType` (schedule_meeting, create_visualization, or null/other).
7. **Emit**: Pass the structured payload `{ intent, normalizedText, speaker, originalSnippet }` to the Automation Orchestrator (M4).
## Outputs

| Output | Description |
|--------|-------------|
| Intent Payload | Structured object with intent type, normalized text, speaker, and original snippet |
| Null | If the chunk does not complete a command frame |

## Limitations

- Intent classification uses simple keyword matching, not semantic understanding â€” "Can someone make a graph?" would not be recognized unless it contains exact keywords.
- No buffer timeout â€” if a user says "Hey NeuroNotes" but never says "Over," the buffer stays open indefinitely (memory leak).
- Only one buffer per meeting â€” nested or overlapping commands are not supported.
- Hardcoded wake words ("Hey NeuroNotes") and end marker ("Over") with no workspace-level customization.
- The entire module is currently embedded inside `AutomationService` rather than existing as an independent component.
- Common speech-to-text transcription errors for "NeuroNotes" (e.g., "neuro nodes", "in your notes") are handled via flexible Regex, but edge cases exist.
