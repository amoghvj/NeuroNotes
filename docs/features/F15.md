# Feature Overview

| Field | Value |
|-------|-------|
| **Feature ID** | F15 |
| **Feature Name** | Demo Mode |
| **Status** | Implemented |
| **Primary Modules** | M5 (Intelligence Core), X1 (Configuration & Environment) |

## Purpose

Allows the entire NeuroNotes application to be demonstrated or tested without requiring active network connections to external AI providers or valid API keys. It ensures that presentations and development work are not blocked by API rate limits, costs, or outages.

## User Journey

1. The developer or presenter sets the `DEMO_MODE=true` environment variable before starting the backend server.
2. The user interacts with the application normally (e.g., asking chat questions, generating summaries, creating visualizations).
3. Instead of waiting for LLM processing, the system instantly returns realistic, hardcoded responses.
4. The UI behavior remains identical to production, providing a seamless demonstration experience.
## Functional Behaviour

The `LLMService` (M5) checks the `DEMO_MODE` configuration flag at the beginning of its primary generation functions. If the flag is true, it intercepts the request and bypasses the external API call. Instead, it inspects the prompt or intended action to determine the appropriate fallback. It returns pre-defined, hardcoded responses formatted exactly as the LLM would (e.g., JSON objects for summaries and visualizations, markdown for chat). This fallback mechanism covers summary generation, chat slash commands, visual generation, and intent parameter extraction.

## Inputs

| Input | Source | Description |
|-------|--------|-------------|
| `DEMO_MODE` flag | X1 (Configuration) | Environment variable `true`/`false` |
| Prompt/Context | Internal Modules | Used to select the correct hardcoded mock |

## Processing

1. **Intercept**: M5 method is called. Check `isDemoMode()`.
2. **Select Mock**: Determine which method was called (e.g., `generateSummary`, `handleChatCommand`).
3. **Return Mock**: Instantly return the corresponding hardcoded response object or string without making any HTTP request.
## Outputs

| Output | Description |
|--------|-------------|
| Mock Response | Hardcoded JSON or text matching the expected LLM output schema |

## Dependencies

| Module | Role |
|--------|------|
| M5 (Intelligence Core) | Implements the fallback logic and stores the mock data |
| X1 (Configuration) | Exposes the feature flag |

## External Dependencies

None. Demo Mode specifically bypasses external dependencies.

## Data Created Or Modified

No specific data is created by this feature itself; it just mocks the data generation for other features.

## Limitations

- Mock data is static; it does not adapt to the actual simulated transcript being injected, which can cause confusing demos where the summary doesn't match the spoken words.
- All mock data is hardcoded directly inside `LLMService.js`, polluting the core logic with large JSON blocks.
- It is a global flag requiring a server restart; it cannot be toggled per-request or from the UI.
