## Purpose

Provides a central hub for users to view an aggregate overview of their activity and access past meeting records. It turns isolated meetings into a searchable, reviewable history, allowing users to track overall productivity and retrieve past insights.

## User Journey

1. The user logs into the NeuroNotes application and lands on the Dashboard.
2. The Dashboard displays top-level statistics (e.g., total meetings, meetings this week, overall productivity score).
3. The user navigates to the Meetings History view.
4. A chronological list of all past, completed meetings is displayed.
5. The user clicks on a past meeting to open its detailed view, revealing its transcript, summary, action items, and insights.
## Functional Behaviour

The application state (M12a) fetches the list of all meetings from the backend (`GET /api/meetings`) upon initialization or view navigation. The frontend calculates dashboard statistics based on this meeting array (e.g., filtering for meetings within the current week, aggregating action item completion rates if available). The Meetings History view renders the array as a list of selectable cards. Clicking a card updates the `activeMeetingId` in the application state and navigates to the detail view, loading the specific meeting's data.

## Inputs

| Input | Source | Description |

## Processing

1. **Fetch**: Frontend calls the backend API to retrieve all meetings.
2. **Transform**: Backend transforms MongoDB documents to match the frontend TypeScript interface.
3. **Store**: Frontend saves the meeting array in global `AppContext`.
4. **Compute Stats**: Frontend dynamically calculates dashboard metrics from the stored array.
5. **Render**: Frontend displays the stats on the Dashboard and the list on the Meetings History view.

## Data Created Or Modified

No data is created or modified by this feature; it is purely read-only presentation.

## Limitations

- The backend currently returns *all* meetings in a single array; there is no pagination or limit, which will cause performance issues as the history grows.
- There is no search or filtering functionality to find specific past meetings by title, participant, or date range.
- Dashboard statistics are computed entirely on the frontend, which requires loading all historical data into the client's memory.
